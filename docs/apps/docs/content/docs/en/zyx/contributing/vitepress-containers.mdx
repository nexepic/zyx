# VitePress Custom Containers Guide

This guide explains how to use VitePress custom containers and styling elements to enhance documentation readability and visual appeal.

## Built-in Containers

VitePress provides several built-in custom containers using the `:::` syntax.

### Info Container

Use for supplementary information, background details, or explanations.

```markdown
::: info Container Title
This is informational content. Can include **bold**, *italic*, `code`, and other Markdown elements.
:::
```

**Rendered output:**

::: info Container Title
This is informational content. Can include **bold**, *italic*, `code`, and other Markdown elements.
:::

**Use cases:**
- Provide background information
- Explain technical concepts
- Add supplementary details

### Tip Container

Use for helpful suggestions, best practices, or optimization tips.

```markdown
::: tip Tip Title
This is a helpful tip or suggestion.
:::
```

**Rendered output:**

::: tip Tip Title
This is a helpful tip or suggestion.
:::

**Use cases:**
- Performance optimization suggestions
- Best practice recommendations
- Shortcut explanations

### Warning Container

Use to alert users about potential issues or important considerations.

```markdown
::: warning Warning Title
This is a warning that users should be aware of.
:::
```

**Rendered output:**

::: warning Warning Title
This is a warning that users should be aware of.
:::

**Use cases:**
- Deprecation warnings
- Compatibility issues
- Configuration considerations

### Danger Container

Use to emphasize serious risks or critical limitations.

```markdown
::: danger Danger Title
This is a dangerous operation or serious limitation.
:::
```

**Rendered output:**

::: danger Danger Title
This is a dangerous operation or serious limitation.
:::

**Use cases:**
- Data loss risks
- Security vulnerability warnings
- Irreversible operation alerts

### Details Container

Use for collapsible detailed content, keeping documentation concise while providing depth.

```markdown
::: details Collapsible Title
Click to expand for detailed content.

This can contain any Markdown content.
:::
```

**Use cases:**
- Detailed technical specifications
- Optional advanced configurations
- Extended reading material

## Badges

While VitePress doesn't have built-in badge components, you can create them using Markdown and custom styling.

### Checkbox Lists

Use for listing completed features or prerequisites:

```markdown
::: tip Prerequisites
Before you begin, make sure you have:
- ✅ Built the ZYX project
- ✅ Generated the `zyx-cli` executable
- ✅ Basic understanding of Cypher query language
:::
```

**Rendered output:**

::: tip Prerequisites
Before you begin, make sure you have:
- ✅ Built the ZYX project
- ✅ Generated the `zyx-cli` executable
- ✅ Basic understanding of Cypher query language
:::

## Best Practices

### 1. Choose the Right Container Type

| Container Type | Use Case | Example |
|---------------|----------|---------|
| `info` | Neutral information, background | Algorithm explanations |
| `tip` | Helpful suggestions, optimizations | Performance tips |
| `warning` | Important considerations | Deprecation warnings |
| `danger` | Serious risks, critical limitations | Data loss risks |
| `details` | Optional detailed content | Extended configuration |

### 2. Keep Container Content Concise

Container content should be brief and to the point. Avoid placing large amounts of text in containers.

**Good practice:**

```markdown
::: info What is a Transaction?
A transaction is a group of database operations executed as a single logical unit.
:::
```

**Bad practice:**

```markdown
::: info
(500 words of detailed explanation here...)
:::
```

### 3. Use Meaningful Titles

Container titles should clearly indicate the nature of the content.

**Good titles:**
- "Prerequisites"
- "Performance Optimization Tips"
- "Known Limitations"

**Bad titles:**
- "Note"
- "Important"
- "Information"

### 4. Nest Containers Appropriately

You can use other Markdown elements inside containers, including lists, code blocks, links, etc.

```markdown
::: tip Configuration Recommendations
Adjust these parameters based on your use case:

1. **High Recall Priority**: Increase `beamWidth` and `maxDegree`
2. **Speed Priority**: Decrease `beamWidth` and `maxDegree`
3. **Memory Constrained**: Decrease `maxDegree`, use more aggressive PQ

See [Parameter Tuning](#parameter-tuning) section for more details.
:::
```

### 5. Avoid Overuse

Don't wrap every paragraph in a container. Containers should highlight important information, not decorate all content.

**Good usage:**
- Key concept explanations
- Important warnings
- Helpful suggestions

**Bad usage:**
- Ordinary descriptive text
- Routine code examples
- Standard section headings

## Containers in Code Examples

Use containers to emphasize key points in code explanations:

```markdown
### Insert Vector

Inserts a new vector into the index and builds graph connections:

```cpp
void insert(int64_t nodeId, const std::vector<float>& vec) {
    // 1. Validate dimension
    if (vec.size() != config_.dim) {
        throw std::invalid_argument("Dimension mismatch");
    }

    // 2. Auto-train PQ if threshold reached
    if (!isPQTrained() && cachedCount_++ == config_.autoTrainThreshold) {
        auto samples = sampleVectors(config_.autoTrainThreshold);
        train(samples);
    }

    // ... more code
}
```

::: tip Complexity Analysis
**Time Complexity**: O(beamWidth × maxDegree × dim)
**Space Complexity**: O(maxDegree)
:::
```

## Common Patterns

### Pattern 1: Prerequisite Checklist

```markdown
::: tip Prerequisites
Before you begin, make sure you have:
- ✅ Installed all dependencies
- ✅ Configured environment variables
- ✅ Read the security notes
:::
```

### Pattern 2: Configuration Details

```markdown
::: details Configuration Details
- **param1**: Parameter description
- **param2**: Parameter description
- **param3**: Parameter description
:::
```

### Pattern 3: Best Practices List

```markdown
::: tip Performance Optimization
1. **Tip 1**: Detailed explanation
2. **Tip 2**: Detailed explanation
3. **Tip 3**: Detailed explanation
:::
```

### Pattern 4: Limitations and Considerations

```markdown
::: warning Known Limitations
- **Limitation 1**: Description
- **Limitation 2**: Description
- **Limitation 3**: Description
:::
```

### Pattern 5: Step-by-Step Instructions

```markdown
::: info Implementation Steps
1. **Step 1**: Description
2. **Step 2**: Description
3. **Step 3**: Description

Each step should...
:::
```
