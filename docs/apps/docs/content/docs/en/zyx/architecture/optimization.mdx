# Optimization Strategies

This guide covers performance optimization techniques for ZYX graph database.

## Query Optimization

### Use Specific Patterns

```cypher
-- GOOD: Start with selective nodes
MATCH (n:Person {name: 'Alice'})-[:KNOWS]->(friend)
RETURN friend

-- AVOID: Too broad patterns
MATCH (n)-[:KNOWS]->(friend)
RETURN n, friend
```

### Filter Early

```cypher
-- GOOD: Filter in WHERE clause
MATCH (n:Person)
WHERE n.age > 30
MATCH (n)-[:KNOWS]->(b)
RETURN b

-- AVOID: Filter after expensive operations
MATCH (n:Person)-[:KNOWS]->(b)
WITH n, b
WHERE n.age > 30
RETURN b
```

### Use Indexes

```cypher
-- Create indexes on frequently queried properties
CREATE INDEX ON :Person(email)
CREATE INDEX ON :Person(name)
CREATE INDEX ON :Person(age)

-- Query will use index
MATCH (n:Person {email: 'alice@example.com'})
RETURN n
```

### Limit Result Sets

```cypher
-- Always use LIMIT when exploring
MATCH (n:Person)
RETURN n
LIMIT 100

-- Use SKIP for pagination
MATCH (n:Person)
RETURN n
ORDER BY n.name
SKIP 100 LIMIT 50
```

## Schema Optimization

### Label Design

```cypher
-- GOOD: Specific labels for different entity types
MATCH (n:Customer)-[:PURCHASED]->(p:Product)
RETURN n, p

-- AVOID: Generic labels
MATCH (n:Entity)-[:RELATED]->(p:Entity)
WHERE n.type = 'Customer' AND p.type = 'Product'
RETURN n, p
```

### Property Optimization

```cypher
-- Store properties efficiently
MATCH (n:Person)
CREATE (n)-[:HAS_METRICS]->(m:Metrics {
  pageViews: 1000,
  uniqueVisitors: 500,
  bounceRate: 0.45
})

-- AVOID: Too many properties on single node
MATCH (n:Person)
SET n.pageViews1 = 1000,
    n.pageViews2 = 2000,
    n.pageViews3 = 3000,
    -- ... many more properties
```

## Transaction Optimization

### Keep Transactions Short

```cpp
// GOOD: Small, focused transactions
auto tx = db->beginTransaction();
tx->execute("CREATE (p:Person {name: 'Alice'})");
tx->commit();

// AVOID: Long-running transactions
auto tx = db->beginTransaction();
tx->execute("CREATE (p:Person {name: 'Bob'})");
// ... wait for user input ...
tx->commit();
```

### Batch Operations

```cpp
// Process in batches
const size_t BATCH_SIZE = 1000;
for (size_t i = 0; i < totalRecords; i += BATCH_SIZE) {
    auto tx = db->beginTransaction();

    std::string query = "UNWIND range($start, $end) AS id "
                        "CREATE (p:Person {id: id})";

    Parameters params;
    params.add("start", i);
    params.add("end", std::min(i + BATCH_SIZE, totalRecords));

    tx->execute(query, params);
    tx->commit();
}
```

## Storage Optimization

### Compression

```cpp
// Enable compression for large segments
DatabaseConfig config;
config.setCompressionEnabled(true);
config.setCompressionType(CompressionType::ZSTD);
config.setCompressionLevel(3);

auto db = Database::create("/path/to/db", config);
```

### Segment Size Tuning

```cpp
// Tune segment size to workload
DatabaseConfig config;

// For many small entities
config.setSegmentSize(256 * 1024); // 256 KB

// For fewer large entities
config.setSegmentSize(2 * 1024 * 1024); // 2 MB

auto db = Database::create("/path/to/db", config);
```

### Checkpoint Configuration

```cpp
// Optimize checkpoint frequency
WALConfig walConfig;

// For write-heavy workloads
walConfig.setCheckpointInterval(Duration::minutes(1));
walConfig.setMaxWalSize(50 * 1024 * 1024); // 50 MB

// For read-heavy workloads
walConfig.setCheckpointInterval(Duration::minutes(10));
walConfig.setMaxWalSize(200 * 1024 * 1024); // 200 MB
```

### Cache Size Tuning


## Concurrency Optimization

### Connection Pooling

```cpp
class ConnectionPool {
public:
    ConnectionPool(size_t poolSize, const std::string& dbPath) {
        for (size_t i = 0; i < poolSize; ++i) {
            auto db = Database::open(dbPath);
            connections_.push_back(std::move(db));
        }
    }

    Database* getConnection() {
        std::lock_guard<std::mutex> lock(mutex_);
        // Get available connection
        return connections_.back().get();
    }

private:
    std::vector<std::unique_ptr<Database>> connections_;
    std::mutex mutex_;
};
```

### Parallel Queries

```cpp
// Execute independent queries in parallel
auto future1 = std::async(std::launch::async, [&]() {
    return db1->execute("MATCH (n:Person) RETURN count(n)");
});

auto future2 = std::async(std::launch::async, [&]() {
    return db2->execute("MATCH (n:Product) RETURN count(n)");
});

auto result1 = future1.get();
auto result2 = future2.get();
```

## Memory Optimization

### Streaming Large Results

```cpp
// Stream results instead of loading all into memory
auto tx = db->beginTransaction();

auto cursor = tx->executeStream(
    "MATCH (n:Person) RETURN n"
);

while (cursor->hasNext()) {
    auto row = cursor->next();
    // Process row
    delete cursor; // Free memory
}

tx->commit();
```

### Batch Processing

```cpp
// Process in batches to limit memory usage
const size_t BATCH_SIZE = 10000;

size_t offset = 0;
while (true) {
    auto result = tx->execute(
        "MATCH (n:Person) RETURN n SKIP $offset LIMIT $limit",
        {{"offset", offset}, {"limit", BATCH_SIZE}}
    );

    if (result.isEmpty()) break;

    // Process batch
    for (const auto& row : result) {
        processRow(row);
    }

    offset += BATCH_SIZE;
}
```

## Performance Monitoring

### Query Profiling

```cypher
-- Profile query execution
PROFILE MATCH (n:Person)-[:KNOWS]->(m:Person)
RETURN n, m

-- Explain query plan (without executing)
EXPLAIN MATCH (n:Person)-[:KNOWS]->(m:Person)
RETURN n, m
```


## Best Practices Summary

1. **Use indexes**: Create indexes on frequently queried properties
2. **Filter early**: Apply WHERE clauses as early as possible
3. **Limit results**: Always use LIMIT when exploring data
4. **Batch operations**: Group multiple operations into batches
5. **Tune cache size**: Adjust cache to workload characteristics
6. **Monitor performance**: Use PROFILE and statistics regularly
7. **Optimize transactions**: Keep transactions short and focused
8. **Use connection pooling**: For multi-threaded applications
