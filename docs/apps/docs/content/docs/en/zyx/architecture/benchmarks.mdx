# Benchmarks

ZYX ships a standalone benchmark application (`apps/benchmark`) that covers insertion, querying, concurrency, vector retrieval, and optimizer scenarios.

## Covered Scenarios

| Category | What it tests |
|----------|--------------|
| **Insert** | Cypher vs. native API write paths, with and without indexes |
| **Query** | Property index lookups, label scans, full scans |
| **Algo** | Shortest path and other graph algorithms |
| **Vector** | Vector insertion and ANN (approximate nearest neighbor) queries |
| **Concurrency** | Multi-threaded read, write, sorting, and vector search |
| **Optimizer** | Side-by-side comparison of optimization rule effectiveness |

## How to Run

Build the benchmark binary and run it directly:

```bash
./buildDir/apps/benchmark/benchmark
```

The benchmark framework (`BenchmarkFramework`) accepts configuration via `BenchmarkConfig` to control dataset size, thread count, warmup iterations, and measurement duration.

## Interpreting Results

When reading benchmark output, keep these guidelines in mind:

1. **Fix your hardware and compiler flags** — Results vary significantly across CPUs, storage devices, and optimization levels. Always use the same `-O3` release build for comparisons.
2. **Separate cold-start from steady-state** — The first run exercises the page cache and WAL creation; subsequent runs benefit from warm caches. Report both.
3. **Use `PROFILE` and `EXPLAIN`** — Combine benchmark numbers with query plan analysis to understand *why* a query is fast or slow.
4. **Check index hit rates** — A query may appear slow not because of the engine, but because an expected index is missing.
5. **Compare across versions with identical setup** — Same compiler, same data distribution, same hardware. Anything else introduces confounding variables.
