# 基准测试

ZYX 提供独立 benchmark 应用（`apps/benchmark`），包含 **51 个基准测试**，覆盖 7 大类别：插入、查询、算法、向量、并发、事务和存储开销。

## 类别总览

| 类别 | 类型标志 | 测试内容 | 数量 |
|------|----------|----------|------|
| **Insert** | `insert` | Cypher 与 Native API 写入路径（单条/批量），4 种索引模式（无 / 标签 / 属性 / 全部） | 16 |
| **Query** | `query` | 属性索引查找、标签扫描、全扫描，以及优化器效果（谓词下推、AND 拆分、Join 过滤） | 7 |
| **Algo** | `algo` | 在 1000 节点链上执行最短路径（Cypher 和 Native API） | 2 |
| **Vector** | `vector` | 向量插入（128d/768d，有/无 HNSW 索引）和 ANN Top-10 检索 | 5 |
| **Concurrency** | `concurrency` | 单线程 vs 多线程对比：查询、排序、批量插入、PQ 训练、向量检索、并发读取 | 12 |
| **Transaction** | `transaction` | 隐式提交 vs 显式事务 vs 显式批量 — 每次运行插入 1000 个节点 | 3 |
| **Storage** | `storage` | 磁盘存储开销分析：数值、短文本、长文本、混合属性、带索引的图、向量索引 | 6 |

## 构建与运行

构建 benchmark 二进制文件：

```bash
ninja -C buildDir zyx-bench
```

运行所有基准测试：

```bash
./buildDir/apps/benchmark/zyx-bench
```

### 命令行选项

| 参数 | 说明 |
|------|------|
| `-t, --type <type,...>` | 按类别过滤（逗号分隔）。可选：`insert`, `query`, `algo`, `vector`, `concurrency`, `storage`, `transaction`, `all` |
| `-b, --bench <id,...>` | 按精确 ID 运行特定测试 |
| `--list` | 打印所有类别和测试 ID 后退出 |
| `--list-types` | 仅打印 7 个类别名后退出 |

类型别名：`write`/`writes` 对应 insert，`read`/`reads` 对应 query，`algorithm`/`algorithms` 对应 algo，`vec` 对应 vector，`conc`/`parallel` 对应 concurrency，`footprint` 对应 storage，`txn` 对应 transaction。

### 示例

```bash
# 仅运行插入测试
./buildDir/apps/benchmark/zyx-bench --type insert

# 运行查询和向量测试
./buildDir/apps/benchmark/zyx-bench --type query,vector

# 按 ID 运行特定测试
./buildDir/apps/benchmark/zyx-bench --bench query.cypher.property_index

# 列出所有可用测试
./buildDir/apps/benchmark/zyx-bench --list
```

## 指标说明

延迟/吞吐量测试报告四列指标：

| 指标 | 说明 |
|------|------|
| **Ops/Sec** | 吞吐量（每秒操作数）。批量测试报告每条目吞吐量。 |
| **Avg(us)** | 每次操作的平均延迟（微秒） |
| **P99(us)** | 按排序的逐次耗时计算的第 99 百分位延迟 |
| **Mem(KB)** | 峰值 RSS，通过 `getrusage`（macOS/Linux）或 `GetProcessMemoryInfo`（Windows）获取 |

每个测试还会报告**阶段耗时分解**（`[Stage avg ms/op]`），按开销降序展示查询引擎各阶段（解析、规划、执行等）的平均耗时。

存储测试使用不同的输出格式：`Nodes | Edges | Props | DB(MB) | WAL(MB) | Total(MB) | Bytes/Node`，并将结果导出到 `/tmp/zyx_storage_profile.csv`。

## 测试清单

### 插入测试

4 种索引模式（`none` / `label` / `prop` / `all`）× 4 种变体 = 16 个测试。

| ID 模式 | 测试内容 |
|---------|----------|
| `insert.cypher.single.<mode>` | 通过 Cypher 每次 `CREATE` 一个节点 |
| `insert.native.single.<mode>` | 通过 C++ API 每次 `db.createNode()` 一个节点 |
| `insert.cypher.batch.<mode>` | 一条多节点 `CREATE` 语句批量创建 |
| `insert.native.batch.<mode>` | `db.createNodes()` 使用预分配数据批量创建 |

### 查询测试

准备数据：500 个 User 节点 + 2000 个噪声节点（Product/Order/Category/Log）。

| ID | 测试内容 |
|----|----------|
| `query.cypher.property_index` | `uid` 属性索引查找（100 次迭代） |
| `query.cypher.label_scan` | 无属性索引的标签扫描查找（20 次迭代） |
| `query.cypher.full_scan` | 无任何索引的全扫描（5 次迭代） |
| `query.optimizer.filter_pushdown` | 优化器将 WHERE 子句下推到索引扫描 |
| `query.optimizer.inline_baseline` | 内联谓词基线（无需优化器变换） |
| `query.optimizer.and_split` | AND 条件在过滤阶段拆分 |
| `query.optimizer.join_filter` | 跨 Join 过滤下推，减少笛卡尔积 |

### 算法测试

均在 1000 节点 `NEXT` 边线性链上运行。

| ID | 测试内容 |
|----|----------|
| `algo.cypher.shortest_path` | 通过 Cypher `CALL algo.shortestPath(startId, endId)` |
| `algo.native.shortest_path` | 通过 Native API `db.getShortestPath(startId, endId)` |

### 向量测试

| ID | 测试内容 |
|----|----------|
| `vector.insert.no_index.128d` | 128 维向量插入（无 HNSW 索引） |
| `vector.insert.indexed.128d` | 128 维向量插入（有 HNSW 索引） |
| `vector.insert.indexed.768d` | 768 维向量插入（有 HNSW 索引） |
| `vector.search.128d.top10` | 500 × 128d 向量 ANN Top-10 检索 |
| `vector.search.768d.top10` | 500 × 768d 向量 ANN Top-10 检索 |

### 事务测试

每次运行均插入 1000 个节点。

| ID | 测试内容 |
|----|----------|
| `transaction.implicit_insert` | 1000 条独立 `CREATE` 语句，逐条自动提交 |
| `transaction.explicit_insert` | 一个显式事务内 1000 条独立 `CREATE` 语句 |
| `transaction.explicit_batch_insert` | 一个显式事务内一条多节点 `CREATE` 语句 |

### 并发测试

每个场景有 `single_thread` 和 `multi_thread`（使用 `hardware_concurrency()` 线程）两个变体。数据规模：10000 个节点，5 次迭代。

| ID 对 | 测试内容 |
|--------|----------|
| `concurrency.query.*` | NodeScan + Filter 算子并行化 |
| `concurrency.sort.*` | 10000 节点上的 SortOperator 并行化 |
| `concurrency.batch_insert.*` | `FileStorage::save()` 并行准备 |
| `concurrency.pq_train.*` | ProductQuantizer 并行子空间训练（1000 × 128d） |
| `concurrency.vector_search.*` | DiskANN 重排序 + PQ 距离表并行化（1000 × 768d） |
| `concurrency.read.*` | 通过 `std::thread` 测试快照隔离下的并发读取 |

### 存储测试

调用 `db.save()` 后测量磁盘占用。

| ID | 创建的数据 |
|----|-----------|
| `storage.numeric` | 5000 个节点，`uid: int64` |
| `storage.short_text` | 5000 个节点，`uid` + 32 字符 `name` |
| `storage.long_text` | 2000 个节点，`uid` + 512 字符 `bio` |
| `storage.mixed` | 5000 个节点，6 个混合类型属性（bool/int/double/string/list） |
| `storage.graph_indexes` | 3000 节点 + 6000 边 + 标签索引和属性索引 |
| `storage.vector_index` | 1200 个节点，128 维向量 + HNSW 索引 |

## 结果解读

1. **固定硬件和编译选项** — 结果因 CPU、存储设备和优化级别而异。对比时始终使用相同的 `-O3` release 构建。
2. **区分冷启动和稳态** — 首次运行会填充页缓存和创建 WAL；后续运行受益于热缓存。两者都应报告。
3. **使用 `PROFILE` 和 `EXPLAIN`** — 将 benchmark 数值与查询计划分析结合，理解查询快或慢的原因。
4. **检查索引命中率** — 查询可能不是因为引擎慢，而是缺少预期索引。
5. **跨版本对比时保持完全相同的配置** — 同一编译器、同一数据分布、同一硬件。
